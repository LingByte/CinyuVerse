# 超长篇小说引擎：原理与实现细节

「超长」的本质问题是：**模型上下文有限，而小说信息无限**。工程解法是 **压缩（摘要）+ 索引（检索）+ 约束（规则与 Canon）+ 过程控制（流水线与校验）** 的组合，而不是单次 prompt 塞全书。

---

## 1. 分层叙事模型（建议必选）

将信息分为不同时间尺度与粒度，写入不同存储，并在生成每一章时 **按预算拼装**：

| 层级 | 内容示例 | 更新频率 | 用途 |
|------|-----------|----------|------|
| L0 全书梗概 | 主题、结局走向（可模糊） | 低频 | 防跑题 |
| L1 卷摘要 | 本卷主线、主要转折 | 每卷或每 N 章 | 中观结构 |
| L2 章摘要 | 本章事件、人物关系变化 | 每章 | 短期连贯 |
| L3 场景块 | 对话与描写原文片段 | 每场景 | 细节与文风 |
| L4 结构化 Canon | 角色生日、武器参数、组织规则 | 变更时版本化 | 防设定矛盾 |

**实现要点**

- 摘要不是一次性的：可采用「滑动窗口摘要」或「增量摘要」（新章生成后更新父级摘要队列中的待处理项）。
- **摘要与原文分离存储**：摘要用于检索与上下文；原文用于展示、导出与法律意义上的「作品版本」。

---

## 2. 章节生成流水线（推荐状态机）

单次「写一章」在内部拆为可观测、可重试的子步骤：

1. **Plan（可选）**：根据大纲与本卷目标产出本章节拍表（beats）：场景列表、冲突、伏笔埋入/回收标记。
2. **Retrieve**：用本章涉及实体（角色 ID、地点 ID、关键词）从向量库与 Canon 拉取 topK 证据片段。
3. **Compose**：拼装系统提示 + L0～L2 + 证据片段 + 上一章结尾 + 用户指令 → 调用模型写正文。
4. **Extract**：让模型或规则抽取：新事实、新人物状态、时间推进（可用 JSON schema 约束）。
5. **Validate**：规则引擎 + 可选二次模型做「一致性裁判」；失败则 **局部重写** 或整章重试（设上限）。
6. **Persist**：事务写入章节正文、摘要、记忆块、时间线；更新检索索引。
7. **Postprocess**：敏感词、格式、章节标题规范化。

**失败策略**

- 子步骤失败应 **可恢复**：记录 checkpoint，避免半章写入导致重复发表。
- 对模型超时：指数退避；对内容校验失败：优先改写段落而非从头生成以省成本。

---

## 3. 记忆块（MemoryChunk）设计

每条记忆建议包含：

- `work_id`, `source_chapter_id`, `source_span`（起止 offset 或 scene_id）
- `entity_ids`（角色/地点/物品）
- `kind`：`fact` | `relationship` | `foreshadow` | `style_note`
- `text`：短句事实描述（便于 embedding）
- `confidence`：抽取模型置信度或人工确认标记
- `valid_until`：可被后续事件撤销（例如「某角色以为 A 死了」）

**为何需要 kind**

- 检索时可按类型加权：写动作戏少拉 `style_note`；回收伏笔多拉 `foreshadow`。

---

## 4. 向量检索与混合检索

**分块策略**

- 按场景或自然段切块，块大小以 embedding 模型上限为准；块间保留少量重叠避免切断语义。
- 对长对话按「轮次」切，不要机械按字数切。

**查询构造**

- 除本章写作提示外，加入 **实体规范化表**（角色名别称映射），减少检索漏召。
- **Hybrid**：BM25（关键词）+ 向量；长书中专有名词多，纯向量易漏。

**重排序（可选）**

- 用小模型或交叉编码器对 topK 再排序，提高「真正相关」片段的排名。

---

## 5. 一致性与矛盾处理

**Canon 表**

- 对「硬设定」使用结构化字段 + 版本号；正文不得与 Canon 冲突。
- 若正文必须突破（例如作者刻意吃书），走 **Canon 变更申请**：记录理由、影响章节范围、级联标记下游 stale。

**轻量规则引擎示例**

- 时间单调：事件时间戳不得倒流（除非明确闪回结构且在元数据中标注）。
- 称谓表：第一人称视角下「我」指代角色 ID 锁定。

**裁判模型（可选）**

- 输入：Canon 片段 + 新章 + 检索证据 → 输出：矛盾列表与严重度。高严重度触发重写 prompt，附带矛盾说明。

---

## 6. 视角、文体与长篇疲劳

**视角锁定**

- 在系统提示中固定 POV 角色 ID；检索时过滤非该视角应知信息（防止「全知泄露」）。

**文风漂移**

- 维护短 `style_note`（不超过几百字），从用户认可的章节中提炼；定期刷新避免陈旧。
- 温度与 top_p 在长篇中宜 **略保守**，牺牲一点花哨换稳定。

---

## 7. 超长作品的「结构工具」

仅堆字数容易水；平台可提供：

- **情节线（Plotline）**：主线/副线，每线有关键节点与完成度。
- **节拍表**：每章关联 beats，生成前检查未完成的伏笔。
- **角色弧**：每个角色有目标、障碍、转变阶段，用于 Plan 阶段约束。

这些结构存在数据库中，由引擎读写；模型负责润色与展开，不负责「记住整张表」。

---

## 8. 测试与回归策略

**Fixture 驱动**

- 维护小型虚构世界观（10 角色、固定年表），自动生成 30～50 章压力测试脚本：每章后跑一致性检查。

**指标**

- 检索召回：人工标注「本章应引用事实」是否出现在 topK。
- 矛盾率：裁判模型或人工抽检统计每万字矛盾条数。
- 重复率：n-gram 或 embedding 相似度检测「自我抄袭」段落。

---

## 9. 与「一次生成全书」的对比

| 方式 | 优点 | 缺点 |
|------|------|------|
| 一次长输出 | 实现简单 | 超出窗口后信息丢失；难以修订 |
| 分章 + 记忆 + 检索（本方案） | 可扩展、可人机协同 | 工程复杂度高 |

超长篇小说平台应选择后者，并把复杂度封装在引擎内，对用户呈现「持续写下去」的体验。
